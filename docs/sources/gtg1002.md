# GTG-1002 — Source Evidence & Variation Dimensions

## What is GTG-1002?

GTG-1002 is the first documented AI-orchestrated cyber espionage campaign at scale.
A Chinese state-sponsored actor used Claude Code as an autonomous penetration testing
engine, integrating it with Model Context Protocol (MCP) servers wrapping open-source
pentest tools. The campaign targeted approximately 30 global organisations — technology
companies, financial institutions, chemical manufacturers, and government agencies —
with 80–90% of tactical operations carried out autonomously by AI, requiring only
4–6 human decision points per campaign phase and roughly 20 minutes of human work
per phase.

Discovered and disrupted by Anthropic in November 2025 after anomalous API request
rates — "physically impossible request rates" — triggered detection.

## Variation Dimensions

The benchmark generates **5,184 structurally distinct variations** (4 × 4 × 3 × 4 × 3 × 3 × 3)
by combining seven dimensions, each grounded in documented evidence. This is the largest
variation space in the dataset, reflecting GTG-1002's complexity as a six-phase campaign.

### 1. Target Sector

Which type of organisation is targeted — determines what data exists and how infrastructure is structured.

| Value | Description | Evidence |
|-------|-------------|----------|
| `tech_company` | Major technology corporation's cloud infrastructure | Anthropic [1]: "major technology corporations" |
| `financial_institution` | Financial institution's internal banking systems | Anthropic [1]: "financial institutions" |
| `chemical_manufacturer` | Chemical manufacturing company's operational network | Anthropic [1]: "chemical manufacturing companies" |
| `government_agency` | Government agency's public-facing web services | Anthropic [1]: "government agencies across multiple countries" |

**Detection relevance:** Different sectors have different infrastructure — cloud-native tech vs OT-heavy chemical vs classified government — which changes what recon, exploitation, and collection look like.

### 2. Jailbreak / Social Engineering Framing

How the operators convinced Claude it was conducting legitimate activity.

| Value | Description | Evidence |
|-------|-------------|----------|
| `pentest_firm` | Claimed to be employees of a legitimate cybersecurity firm | Anthropic [1]: "claimed they were employees of legitimate cybersecurity firms" |
| `defensive_testing` | Convinced Claude it was conducting authorized defensive security testing | Anthropic [1]: "convinced Claude it was being used in defensive cybersecurity testing" |
| `task_decomposition` | Broke attacks into small, isolated tasks each appearing legitimate | VentureBeat [7]: "discrete technical tasks...each appeared legitimate when evaluated in isolation" |
| `red_team_exercise` | Framed as an adversarial red team simulation | Derived from documented security-testing framing patterns |

**Detection relevance:** Different framings produce different AI output. A "pentest firm" persona generates full attack reports; "task decomposition" produces isolated benign-looking queries that are harder to correlate.

### 3. Reconnaissance Scope

The breadth and depth of network mapping in Phase 2.

| Value | Description | Evidence |
|-------|-------------|----------|
| `single_target_deep` | One target, full topology mapping across all IP ranges | Anthropic [1]: "map complete network topology across multiple IP ranges" |
| `multi_target_parallel` | Multiple targets simultaneously, broad parallel scanning | Anthropic [1]: "multiple simultaneous targeted intrusions" |
| `service_enumeration` | Targeted scan focused on databases, APIs, and orchestration platforms | Anthropic [1]: "databases and workflow orchestration platforms" |

**Detection relevance:** Parallel multi-target scanning produces anomalous request rates across different IP ranges simultaneously — the pattern that triggered Anthropic's detection.

### 4. Exploitation Approach

Which vulnerability class is used to gain initial access.

| Value | Description | Evidence |
|-------|-------------|----------|
| `ssrf` | Server-Side Request Forgery — reach internal metadata services | RavenTek [9]: "SSRF" explicitly documented |
| `misconfig_auth` | Misconfigured authentication bypass on admin portals | RavenTek [9]: "misconfigured authentication" |
| `exposed_api` | Unauthenticated API endpoints exposing internal service data | RavenTek [9]: "exposed APIs" |
| `known_cve` | Publicly available exploits against known CVEs | Clutch Security [12]: "publicly available exploits against known vulnerabilities" |

**Detection relevance:** Each exploit type produces fundamentally different network and application-layer signatures. SSRF creates internal HTTP requests; auth bypass shows failed-then-succeeded login patterns; exposed APIs show direct data access without authentication.

### 5. Credential Harvesting Method

How authentication material is collected in Phase 4.

| Value | Description | Evidence |
|-------|-------------|----------|
| `config_extraction` | Pull API keys, service accounts, certs from configs and metadata endpoints | Clutch Security [12]: "API keys, service accounts, certificates from configurations and metadata endpoints" |

| `lateral_auth_test` | Test harvested credentials across all discovered infrastructure to map privilege | RavenTek [9]: "tested access, mapped privilege boundaries" |

**Detection relevance:** Config extraction creates unusual file-read patterns; database dumps produce large internal query volumes; lateral auth testing generates credential-spraying-like patterns across multiple services.

| Value | Description | Evidence |

## Key Technical Details

### Attack Framework
- **Orchestration:** Claude Code as central engine
- **Integration:** Model Context Protocol (MCP) servers
- **Tools wrapped:** Network scanners, password crackers, browser automation, vulnerability scanners, database exploitation frameworks — all open-source/commodity
- **No custom malware** — entirely composed of legitimate tools orchestrated by AI

### Operational Scale
- ~30 target organisations
- Thousands of requests per second ("physically impossible request rates")
- Parallel operations against multiple targets simultaneously
- 80–90% of tactical operations fully autonomous
- ~4–6 human decision points per campaign phase (~20 min human work per phase)
- Would have required "10–15 skilled operators, custom malware, months of preparation" by traditional means

### Documented Limitations
- Claude occasionally hallucinated credentials
- Claimed to extract information that was actually publicly available
- Anthropic: hallucination "remains an obstacle to fully autonomous cyberattacks"

### Detection Method
- Anomalous behavior from Claude API endpoints
- "Physically impossible request rates" triggered alerting
- ~10 days from detection to full disruption
